# Reflexão Pessoal
O que foi difícil?
Acho que a primeira dificuldade foi a criação de arquivos. Tentei criar pelos arquivos normais (explorador do Windows), mas não aparecia pelo GitHub Desktop, então tive que pesquisar e descobri que podia criar também pelo Visual Studio Code, o que finalmente deu certo. Depois disso, uma grande dificuldade foi a nomenclatura nos commits; tive que pesquisar o jeito profissional de fazer, que é requerido pela atividade, e achei essa questão de usar "feat", "docs", "fix", etc. Para finalizar, a organização geral do repositório ainda é uma dificuldade pessoal bem grande; preciso me acostumar com o estilo de como manter um repositório organizado e geri-lo de forma responsável.

O que ficou claro?
Como fazer os commits foi uma coisa que ficou bem clara. É um conceito simples, mas fundamental para um repositório, e creio que ficou bem fixado após essa atividade. Além disso, entendi o conceito de push to origin para mandar o commit para o repositório lá no GitHub. Para finalizar, a importância dos commits locais e a responsabilidade de dar um push foram conceitos bastante usados nesta atividade e, na minha opinião, foram bem lapidados dentro do escopo proposto.

O que ainda é confuso?
Sinceramente, bastante coisa ainda é confusa, pois tenho pouco tempo de programação e menos ainda de uso do GitHub. Literalmente, esse é o meu segundo projeto e o primeiro com impacto real, logo, a profundidade e a noção de GitHub que devo ter precisam ser altas. Primeiramente, não entendi direito a importância dos Pull Requests e quando devemos utilizá-los; creio que seja por segurança, mas não fixei direito a importância dessa função. Além disso, a questão de pegar uma branch e fazer um merge (ou seja, uma junção de duas branches) ainda me parece abstrata; consigo ver poucos cenários onde isso vai ser usado e não entendo como será útil fora de contextos bem específicos. Para finalizar, a questão da resolução de conflitos é um pouco confusa. A parte do GitHub é tranquila: o programa avisa que haverá um erro se você fizer aquele merge e, quando continuamos, temos que resolver esse conflito para seguir alterando a branch. A parte confusa mesmo é no Visual Studio Code, onde aparecem vários símbolos de "menor que" e "maior que" (<<<<, >>>>) e opções que nem parecem ser selecionáveis. Além disso, a interface está em inglês e temos várias opções de escolha; para achar a certa para o meu caso, tive que pesquisar o que cada termo significava até entender que eu deveria selecionar o "Accept Both Changes" para finalmente resolver o conflito.

Conclusão
Olhando para o resultado final, percebo que o Git é um sistema de controle que exige disciplina. Logo, eu, como desenvolvedor, devo procurar aprender cada dia mais para deixar os meus projetos — inclusive este — em um nível profissional, não só no código, mas também na organização. Ademais, devo pesquisar e aprender mais sobre a plataforma, versionamento e boas práticas para que, nos códigos futuros, eu possa manter um padrão de excelência.
